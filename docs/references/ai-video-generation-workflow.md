# AI Video Generation Workflow

Use this reference for video projects where a meaningful part of the work is
done through external AI image or video models: Flow / Veo / Omni, Seedance,
Kling, ImageGen, or similar tools.

The goal is to reduce live handholding. The human should provide creative
intent and taste. The agent should keep the project organized, run small tests,
track outputs, and promote only validated assets.

## When To Use

Use this workflow when a project involves any of these:

- reference images or keyframes generated by AI
- image-to-video or start/end-frame video generations
- video style transfer or edit modes
- multiple short model outputs stitched into one sequence
- repeated browser downloads from video generation tools
- many visual variants that need review, naming, and assembly

For code-first Remotion work without external model iteration, use the normal
`creating-video` workflow.

## Golden Loop

1. Create the project home before generating assets.
2. Capture a short intake contract.
3. Create a clip ledger and prompt folder.
4. Generate one cheap canary before a batch.
5. Import, normalize, and make review frames for every useful output.
6. Record the verdict and next action in the ledger.
7. Promote keepers into a finalized working set.
8. Update the project handoff before ending the session.

Do not rely on chat memory for current files, prompts, or verdicts.

## Project Layout

For AI-heavy projects, use this layout under `projects/<project-id>/`:

```text
projects/<project-id>/
  clip-ledger.jsonl
  prompts/
    README.md
    prompt-name.md
  raw-imports/
  finalized-working-set/
  review-frames/
  style-reference-stills/
  transition-keyframes/
  drafts/
```

Use `docs/projects/<project-id>/tasks.md` for active execution state and a
handoff doc when the project becomes too large for one session.

## Intake Contract

Keep intake short. If the user is brainstorming, make a reasonable default and
run a canary instead of asking many questions.

Minimum contract:

- project id and title
- target aspect ratio, resolution, fps, and rough duration
- source media and whether it is read-only
- current story shape in 3-7 beats
- model path to test first
- what counts as a keeper
- what must not change

For most horizontal web videos, default to 16:9, 1920x1080, 30fps unless the
user says otherwise.

## Clip Ledger

Every generated or imported candidate should get a ledger entry. JSONL is easy
for agents to append and parse:

```json
{"id":"pixel-world-v03","role":"world_clip","source":"Downloads/example.mp4","model":"Flow / Veo","prompt":"prompts/pixel-world-walk.md","duration_s":8,"resolution":"1280x720","fps":24,"output":"projects/paper-portal/finalized-working-set/03-pixel-world-walks-into-portal-current-candidate.mp4","verdict":"keeper","failure_modes":[],"next_action":"append after pencil portal transition"}
```

Recommended fields:

- `id`
- `role`
- `source`
- `references`
- `model`
- `settings`
- `prompt`
- `duration_s`
- `resolution`
- `fps`
- `output`
- `review_frames`
- `verdict`: `canary`, `keeper`, `reject`, `needs_trim`, `needs_prompt_fix`
- `failure_modes`
- `next_action`

The ledger should answer: what is this file, why does it exist, should we use
it, and what should happen next?

## Prompt Folder

Do not leave successful prompts only in chat. Store reusable prompts in
`projects/<project-id>/prompts/`.

Good prompt files include:

- the positive prompt
- negative / avoid list when useful
- input references used
- target model and mode
- known failure modes
- the best output id from the ledger

Keep model-specific reusable blocks in `docs/references/ai-video-model-playbook.md`
instead of copying large prompt blocks into every project.

## Decision Loop

When the user asks "what do you think?", the agent should usually do this:

1. Name the decision in plain language.
2. Offer 2-3 viable options.
3. Recommend one default with a short reason.
4. Run the smallest useful test.
5. Return evidence: file path, screenshot/contact sheet, and verdict.

Avoid long abstract debate when a canary can answer the question.

## Canary Rules

Use canaries before expensive or multi-output runs.

Good canaries are:

- 2-5 seconds when testing motion
- low or medium quality when testing prompt behavior
- one action only
- one model variable changed at a time
- validated with stills or contact sheets

Promote to full quality only after the canary proves the motion and composition.

## Keyframe Workflow

Use keyframes when exact composition matters.

Rules:

- Keep all keyframes in the same aspect ratio.
- Keep the ground line, horizon, portal, or door aligned.
- Use a clear start image and a clear end image.
- Prefer one major action per clip.
- If an edit needs only a face, arm, or small prop change, use a tight mask.
- Avoid broad edits when character positions must stay fixed.

Before sending start/end frames to a model, visually check that:

- framing matches
- important characters are not too close to crop edges
- the walking path is clear
- the final pose is physically possible
- no accidental handle, seam, gap, or duplicate object creates ambiguity

## Import And Normalize

External generation tools often return different sizes, frame rates, codecs, and
names. Import useful downloads before reviewing them seriously.

Default import flow:

1. Copy raw files into `projects/<project-id>/raw-imports/`.
2. Rename with semantic ids.
3. Run `ffprobe` and record duration, resolution, and fps.
4. Normalize assembly copies to the project target format.
5. Make contact sheets or representative review frames.
6. Update `clip-ledger.jsonl`.
7. Promote only keepers into `finalized-working-set/`.

Do not assemble directly from random `Downloads/` paths.

## Review Frames

For every candidate clip that might be used, generate at least:

- first frame
- mid frame
- final frame
- a contact sheet when motion or style changes matter

Use review frames to catch:

- unwanted crop changes
- characters leaving frame
- extra limbs or duplicate characters
- portal/glow explosions
- style drift
- broken final pose
- jump cuts caused by misaligned keyframes

## Assembly Rules

Assemble keepers only after they have stable names and ledger entries.

Default assembly behavior:

- normalize all clips to the same target format first
- use direct cuts for rough review
- add transitions only after the story order works
- keep transition clips short and purposeful
- use sound effects/music after visual lock unless sound is part of the test

If a join feels abrupt, first check whether the outgoing and incoming frames are
compositionally compatible. Do not hide a broken cut with effects too early.

## Handoff Rules

Before ending a long AI-video session, update:

- `docs/projects/<project-id>/tasks.md`
- a project handoff doc if the state is complex
- `projects/<project-id>/finalized-working-set/README.md`
- `clip-ledger.jsonl`

The handoff should identify:

- current keeper
- current source for style transfer or extension
- current prompts
- rejected model paths and why
- open decisions
- exact next action

## Paper Portal Lessons Generalized

The Paper Portal session produced these durable defaults:

- Short independent clips are more reliable than one long generation.
- Door or portal crossings are useful reset points.
- Start/end image workflows are usually easier to control than open-ended
  video extension.
- Simple character motion works better than complex acting.
- A "surprise" beat can make models overact; use "happy and curious" unless
  the reaction is the main point.
- Side characters need explicit locks or they will move, vanish, or follow the
  hero.
- Style names should be descriptive rather than brand-specific.
- The agent should create screenshots/contact sheets without being asked.
- The agent should import and normalize downloaded outputs immediately.
